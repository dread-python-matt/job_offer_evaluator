**Reviewer role:** Senior QA / Architecture review
**Date:** 2026-06-30
**Branch reviewed:** `mvp` (including uncommitted working-tree changes)
**Scope:** Backend (`app/`, `main.py`, migrations, CI/Docker), security focus, with a skim of the Angular auth surface. Frontend not exhaustively reviewed.

**Build/lint/test status at review time:** `ruff check` → clean; `uv run pytest -q` → **763 passed, 35 skipped** (integration tests self-skip without a DB), 11.7 s.

## 1. Executive summary

This is an **unusually well-engineered codebase** for an MVP. Clean/Hexagonal architecture is applied consistently and correctly, the dependency rule is respected, the authentication subsystem is genuinely production-grade (RFC 9700 refresh rotation with reuse detection, argon2id, timing-attack and enumeration mitigations, CSRF double-submit, token-version revocation), and test coverage is broad and green. There are **no critical or high-severity security vulnerabilities** in the code itself.

The gap between this and "ship to production with confidence" is **not** correctness of the core logic — it is a handful of **operational / deploy-time hardening issues**, some **multi-worker scalability caveats**, and **branch hygiene** (a half-wired feature, superseded modules, and documentation drift from in-flight work).

### Overall verdict

| Dimension | Rating | One-line |
|---|:--:|---|
| Security (code) | **A−** | No injection/authz holes found; strong auth. One deploy-config footgun. |

**Recommendation:** Ship-able after addressing the **High** item (§3) and the multi-worker items (§5) for any deployment with `WORKERS > 1` or an OpenAI admin key. The branch should be cleaned (§6) before it is considered "done".

## 2. What is done well (strengths worth preserving)

These are called out so the remediation below is read in context — the bar here is high.

- **Authentication & session management** (`app/application/auth_use_cases.py`, `app/application/refresh_tokens.py`, `app/presentation/api/auth.py`):
  - Refresh-token **rotation with reuse detection and family revocation** (RFC 9700), refresh tokens stored **SHA-256-hashed only**, opportunistic expiry purge to bound the table.
  - **argon2id** hashing (OWASP default) behind a port.
  - **Login timing-attack mitigation** via a dummy hash (`AuthenticateUserUseCase._dummy_hash`) so a missing email costs the same as a wrong password.
  - **Enumeration-resistant** forgot-password (always 202) and generic login errors.
  - **Single-use reset links** bound to `token_version` (a completed reset invalidates the link even within TTL).
  - **CSRF double-submit** with `secrets.compare_digest` (constant-time); refresh cookie scoped to `/auth`, httpOnly; CSRF cookie deliberately readable.
  - **Instant revocation / logout-everywhere** via `token_version` in the JWT.
- **JWT** (`jwt_token_service.py`): HS256 with an **explicit `algorithms=[...]`** allow-list on decode (no algorithm-confusion), `exp` enforced.
- **Secrets at rest:** provider/admin API keys are **Fernet-encrypted (not hashed)** with the secret held outside the DB, validated against the provider before storage, and only ever surfaced as masked hints.
- **No SQL injection surface:** the entire data layer uses SQLAlchemy expression language with bound parameters (`postgres_offer_repository.py`); user input only ever becomes a bound `LIKE` *value*, never SQL text.
- **DoS-amplification caps at the edge:** `UserProfileSchema` bounds summary length and list sizes; `offers_limit`/`offers_to_score` are bounded (`schemas.py`).
- **Defense-in-depth:** `SecurityHeadersMiddleware` (CSP `default-src 'none'`, `X-Frame-Options: DENY`, nosniff, Referrer-Policy, conditional HSTS); catch-all exception handler returns a generic 500 and never leaks internals (`error_handlers.py`).
- **Fail-fast config validation** exists (`config_validation.py`) — refuses to boot in production with the dev JWT secret, dev Fernet key, non-secure cookies, or wildcard CORS.
- **Observability:** structured JSON logging, per-request correlation id via `contextvar` that survives `await`/task fan-out, query strings omitted from access logs (tokens never logged).
- **Concurrency correctness for usage accounting:** `RequestScopedModelUsageTracker` uses a `ContextVar` so concurrent AI matches can't cross-attribute tokens between tenants.
- **Cost control:** content-addressed AI-score cache, per-key + org-backstop budget gate, client-side Gemini pacing honoring `Retry-After`, and a fail-fast `limit: 0` quota-wall message.
- **Prompt-injection awareness:** the LLM system prompt explicitly designates candidate/offer text as untrusted data, reinforced by a constrained structured output type (`llm_scoring_strategy.py`).
- **Bounded LRU caches** for the per-user AI use cases and model lists (`AiScoringContext`, `CachingUserAvailableModelsProvider`) — eviction is implemented, not forgotten.

## 3. Findings — HIGH

### H-1. Production security hardening is opt-in (`APP_ENV` gate), with publicly-known dev secrets in the repo
**Files:** `app/config_validation.py:57-58`, `app/config.py:15,76,85`

`validate_runtime_config()` returns early unless `APP_ENV == "production"`:

```python
if app_env != "production":
    return
```

`APP_ENV` defaults to `"development"`, and the **dev `JWT_SECRET` and dev `API_KEY_ENCRYPTION_KEY` are committed in `config.py`** as fallbacks. Neither the `Dockerfile` nor `docker-compose.yml` sets `APP_ENV=production`. Consequently, a deployment that simply forgets to set `APP_ENV=production` (or sets it to anything else) will **boot silently with the known-public dev secrets** — at which point anyone with repo access can forge arbitrary session JWTs and decrypt every stored provider/admin API key.

This is a "secure by configuration" rather than "secure by default" posture. The blast radius (full session forgery + key decryption) is severe; the trigger is a single missing env var.

**Recommendation (pick one, ideally both):**
- Invert the default: treat any value other than an explicit `development`/`test` as production-grade, **or** always enforce the JWT/Fernet checks regardless of `APP_ENV` (only relax cookie/CORS strictness for dev).
- Make the dev fallbacks non-functional outside dev: refuse to start if `JWT_SECRET`/`API_KEY_ENCRYPTION_KEY` equal the committed dev constants and `APP_ENV != development`.
- Set `APP_ENV=production` explicitly in the production image/compose and document it as required.

## 4. Findings — MEDIUM

### M-1. Multi-worker Google/Gemini pacing over-paces by a factor of `WORKERS`
**Files:** `main.py:486-501`, `app/config.py:111-114`

The Google rate limiter is a **per-process** dict (`_google_rate_limiters`). With `WORKERS > 1`, each worker paces independently to the model's RPM, so aggregate client-side RPM ≈ `WORKERS × model_rpm` — which **exceeds the provider's per-project cap and trips 429s** exactly when pacing was supposed to prevent them. The README documents the analogous caveat for the *login* throttle and offers a Redis backend for it, but `RATE_LIMITER_BACKEND=redis` does **not** cover Gemini pacing. This is a silent correctness gap for the default provider (Gemini) under horizontal scaling.

**Recommendation:** Either divide each model's RPM budget by `WORKERS`, or move pacing to a shared store (Redis token bucket), or document that Gemini free-tier pacing is only correct at `WORKERS=1`.

### M-2. Unbounded in-memory cache: `_google_rate_limiters`
**File:** `main.py:487-501`

Unlike the carefully bounded LRU caches elsewhere (`AiScoringContext`, `CachingUserAvailableModelsProvider`), `_google_rate_limiters: dict[(user_id, model)]` has **no eviction**. It grows one entry per distinct `(user, model)` for the lifetime of the process — slow unbounded memory growth in a multi-tenant deployment, and inconsistent with the bounded-cache discipline applied two modules over. Looks like an oversight rather than intent.

**Recommendation:** Use the same bounded `OrderedDict` LRU pattern already established in `AiScoringContext`.

### M-3. Dual schema management: repositories run `create_all` DDL at construction *and* Alembic owns the schema
**Files:** all `app/infrastructure/postgres_*_repository.py` (9 call sites), `README.md` "Schema caveat"

Every Postgres repository calls `Base.metadata.create_all(...)` for its own table in `__init__`. The README already flags the resulting footgun: `create_all` will build tables from the ORM models if migrations haven't run, leaving the Alembic version table unstamped and a subsequent `alembic upgrade head` in an inconsistent state; and it never adds columns to a pre-existing table. The Docker `CMD` happens to run `alembic upgrade head` *before* the app constructs repos, so the prod path is currently safe — but this couples adapter instantiation to schema DDL, duplicates the source of truth, and is a latent production incident (e.g. a new replica, a test harness, or anyone running `python main.py` against a fresh DB without migrations).

**Recommendation:** Remove `create_all` from the repositories; rely on Alembic exclusively in all non-test paths. Have the test suite create the schema via a fixture/`Base.metadata.create_all` in `conftest`, not inside production adapters.

### M-4. Half-wired feature on the branch: per-day request limit (dead in production)
**Files:** `app/infrastructure/daily_request_usage_reader.py`, `app/application/api_key_use_cases.py` (`SetDailyRequestLimitUseCase`), `app/application/use_cases.py` (`GetDailyRequestUsageUseCase`), `orm_models.py:189`, `alembic/versions/0017_user_api_key_daily_request_limit.py`

The daily-request-budget subsystem is fully built — ORM column, migration, port methods, domain `DailyRequestStatus`, a reader adapter, and two use cases — and `MatchOffersWithAiUseCase` already supports `daily_request_reader`/`scoring_model` parameters. **But none of it is wired in `main.py`**: `_build_ai_use_case` does not pass a `daily_request_reader`, no route exposes `GetDailyRequestUsageUseCase`/`SetDailyRequestLimitUseCase`, and `grep` for these symbols in `main.py` returns nothing. The migration ships a schema change for a feature that does nothing at runtime.

This is acceptable as in-progress work but **must not reach `main`/production in this state**: it's confusing dead surface area and a schema migration with no behavior.

**Recommendation:** Either finish the wiring (gate + route) or remove the use cases/reader/route-less code until ready (keep the migration only if you intend to keep the column).

### M-5. Composition root has accreted non-trivial logic and per-request factories
**File:** `main.py` (642 lines)

`main.py` has grown well beyond pure wiring: request-scoped factory functions (`_org_spend_use_case_for_request`, `_ai_use_case_for_request`), per-`(user,model)` cache management (`_google_rate_limiter_for`), late-binding closures (`_on_user_keys_changed` referencing `_ai_scoring_context` before it exists), and budget-gate assembly (`_build_budget_gate`). This is the one place the otherwise-clean layering frays — behavior that belongs in named application/infrastructure collaborators lives as module-level functions and mutable dicts in the root.

**Recommendation:** Extract an `AiUseCaseFactory` (owns model→provider resolution, key resolution, the Google limiter cache, budget-gate building) and an `OrgReadoutFactory`. The root then just constructs and binds them. This shrinks the root, makes the closures testable, and removes the forward-reference fragility.

## 5. Findings — LOW

### L-1. Registration leaks account existence (email enumeration)
**File:** `app/presentation/api/auth.py:239-240`

`POST /auth/register` returns **409 "Email already registered"** for a known address. While login and forgot-password are deliberately enumeration-resistant, registration is not, so the registered-email set is discoverable. The per-`(IP,email)` throttle bounds *brute-force* but not *enumeration* (an attacker varies the email; each gets its own bucket, and there is no per-IP global cap). Severity is low (industry-common, and email verification is required), but it is inconsistent with the care taken elsewhere.

**Recommendation:** Consider returning the same 202 regardless and emailing either a confirmation or a "you already have an account" notice; or accept the trade-off and document it.

### L-2. Check-then-act races surface as 500 instead of 4xx
**Files:** `app/application/api_key_use_cases.py:75-76` (`AddApiKeyUseCase`), `app/application/auth_use_cases.py:76-77` (`RegisterUserUseCase`), `app/application/api_key_use_cases.py:124-128` (`SetApiKeyBudgetUseCase` update-then-get)

Uniqueness is checked with `get(...)` then `add(...)`. Two concurrent requests can both pass the check; the DB unique constraint correctly preserves integrity, but the second `INSERT` raises `IntegrityError`, which is **not** translated to `ApiKeyAlreadyExistsError`/`EmailAlreadyRegisteredError` — it bubbles to the catch-all **500** instead of a clean **409**. `SetApiKeyBudgetUseCase` similarly does `update` then `get` and would `NoneType`-crash if the row is deleted in between. Data is never corrupted; only the error contract degrades under a rare race.

**Recommendation:** Catch `IntegrityError` in the repository/use case and map to the domain "already exists" error; guard the post-update `get` against `None`.

### L-3. Redis fixed-window `incr`+`expire` is non-atomic (rare permanent-lockout edge)
**File:** `app/infrastructure/redis_rate_limiter.py:49-54`

`record_failure` does `incr` then, if `count == 1`, `expire`. A crash/network drop between the two leaves a counter with **no TTL** → that `(IP,email)` key never resets → a user can be locked out of login/forgot-password until manual intervention. Low probability, but a known fixed-window-in-Redis pitfall.

**Recommendation:** Make it atomic — a small Lua script (`INCR` + `EXPIRE` on first hit) or `SET key 1 EX <window> NX` then `INCR`.

### L-4. In-memory limiter increment is read-modify-write, not atomic
**File:** `app/infrastructure/in_memory_rate_limiter.py:44-50`

`record_failure` reads `_live_window` then writes `count+1`. The docstring claims correctness "relies only on CPython dict atomicity," but the increment spans two bytecodes, so under threaded concurrency two failures can race and lose one increment (slightly *under*-counting, i.e. allowing a few extra attempts). Acceptable for the documented single-worker scope; worth a more precise comment or a lock.

### L-5. `LIKE` wildcards in user input are not escaped (functional, not security)
**File:** `app/infrastructure/postgres_offer_repository.py:99-126`

Search/location/tech terms are interpolated into `LIKE` patterns without escaping `%`/`_`, so a user searching `c%` matches more than intended. Not an injection (values are bound), purely a search-semantics nicety.

### L-6. OrgSpendBackstop uses a single global limit equal to a per-user default
**Files:** `main.py:311`, `app/infrastructure/org_spend_backstop.py`

When an OpenAI admin key is configured, the org-wide daily spend cap is `DEFAULT_BUDGET_USD` (default **$5**) shared across *all* tenants — so the sum of all users' AI spend blocks everyone at $5/day, and the cap reuses the *single-user* default. Inert by default (Gemini has no admin key), but surprising for OpenAI deployments.

**Recommendation:** Give the org backstop its own explicit env var (e.g. `ORG_DAILY_BUDGET_USD`) decoupled from the per-user default.

## 6. Findings — INFORMATIONAL / Branch hygiene

### I-1. Superseded modules retained (dead in production)
- **Fully unreferenced:** `app/infrastructure/persisting_model_usage_tracker.py` (no imports, no tests).
- **Tested but not wired into production** (superseded by the per-user keyed variants / `RequestScopedModelUsageTracker`): `composite_available_models_provider.py`, `caching_available_models_provider.py`, `logging_model_usage_tracker.py`, `composite_model_usage_tracker.py`.

These cost maintenance and obscure the real wiring. Recommend deletion (with their now-orphaned tests) unless a near-term use is planned.

### I-2. `mypy` referenced as workflow but not a dependency
**Files:** `CLAUDE.md` ("Key Commands" lists `uv run mypy`), `.claude/settings.json` vs `pyproject.toml`

`mypy` is documented as part of the workflow but is **not** in the `dev` dependency group and there is no `[tool.mypy]` config, so `uv run mypy` will not run as documented and type-checking is not enforced in CI. Either add `mypy` (+ config + a CI step) or drop the command from the docs. Given the codebase already annotates public functions thoroughly, adding mypy to CI would be high-value and low-friction.

### I-3. Documentation drift (README behind the working branch)
**File:** `README.md`

The README "Data model" / migrations section lists migrations **`0001`–`0014`** and omits `0015_model_usage_cost`, `0016_openai_admin_key`, `0017_user_api_key_daily_request_limit`, and the `openai_admin_key` table / per-user admin-key feature now present in code. The API table doesn't list the `/api-keys*` and `/admin-key` routes that exist in `routes.py`. Update before release; the README is otherwise the project's standout asset.

### I-4. No `[tool.ruff]` config / line-length policy
`ruff` runs on defaults (E/F subset), so `E501` line-length is not enforced and several long lines exist (e.g. `main.py` dependency-override block). Fine as-is, but pin a config for consistency if the team cares about width.

### I-5. Migrations run in the app start `CMD` (race under >1 replica)
**File:** `Dockerfile:32`

`CMD ["sh","-c","uv run alembic upgrade head && uv run python main.py"]` is correct for a single instance, but scaling to multiple replicas will run concurrent migrations on boot. Move migrations to an init container / one-shot job / release phase for horizontal scaling.

### I-6. No per-user request-rate limit on the expensive AI endpoint
`/offers/match/ai` is bounded *within* a request (`AI_MATCH_CONCURRENCY`) and by budget, but nothing caps a single user's *concurrent* AI-match requests, so fan-out across simultaneous requests is unbounded until the budget gate catches up. Consider a per-user in-flight cap for fairness/DoS resistance.
